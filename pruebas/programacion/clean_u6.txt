2.4.3 Escritura de datos relativa
Como sucedía en el tratamiento secuencial, a la hora de realizar lecturas mediante acceso relativo hay que tener cuidado y usar siempre el método que corresponde al tipo del valor al que se quiere acceder, así como procurar no acceder a posiciones más allá del tamaño real del fichero.
Para el caso del acceso relativo, además, también debe ser muy cuidadoso para indicar cuál debe ser la posición del apuntador. En caso contrario, el programa se comportará de forma errática, ya que se leerán bytes de diferentes valores mezclados. Por ejemplo, si un archivo sólo contiene valores enteros, las únicas posiciones válidas para realizar lecturas son la 0, 4, 8, 12, etc.
El acceso relativo a un archivo evita tener que cargar todos los datos a memoria, por ejemplo a un array, y luego volver a escribirlos si se quieren realizar modificaciones a valores individuales. Estas enmiendas se pueden realizar directamente sobre las posiciones del archivo.
Una vez que el apuntador se encuentra en la posición que se quiere modificar, el proceso de escritura mediante acceso relativo es similar al de lectura, simplemente cambiando la invocación de métodos de lectura por los de escritura (writeInt(...), writeDouble(...), etc.).

Si se derrota al adversario, se otorga una puntuación al jugador. Si el jugador es derrotado, termina la partida. El objetivo final del jugador es sobrevivir diez combates, logrando la máxima puntuación posible en el proceso.

La propagación de errores es el mecanismo mediante el cual un método avisa al código que le ha invocado si ha podido realizar su tarea correctamente o
Cuando se realizan operaciones sobre archivo, pueden suceder una serie de situaciones erróneas. Por ejemplo, intentar leer datos de un archivo inexistente, leer datos cuando ya se ha llegado al final del archivo o de un tipo inesperado, al tener el archivo un formato incorrecto. En Java, estos errores se llaman excepciones, y es necesario controlarlos usando una sentencia try/catch.
En el caso de un programa modular con archivos, es muy importante que, cada vez que se invoca un método donde dentro de su código se tratan archivos, el código que ha llevado a cabo la invocación pueda establecer si todo ha ido bien o no. Por tanto, los métodos que tratan datos en ficheros deben poder avisar si han realizado todas las tareas correctamente o no.
En el caso de métodos que no disponen de ningún parámetro de salida (tipo void), se fuerza a que tengan un parámetro de salida de tipo entero, que servirá exclusivamente para establecer si todo ha ido bien o no. Normalmente se hace que se evalúen a 0 si todo ha ido bien y ­1 si ha habido alguna excepción.
Por ejemplo, mostrando un mensaje de error por pantalla.
Existen distintos mecanismos para llevar a cabo la propagación de errores. Para esta clase se ha usado lo más simple, que es reservar uno de los valores del parámetro de salida para indicar que ha sucedido un error (a menudo se utiliza el valor ­1). Una vez evaluado el método invocado, comprobar si el resultado es el valor reservado o no. Si es así, es que ha habido una excepción dentro del método, y por tanto éste no ha podido llevar a cabo su tarea correctamente. Entonces, es necesario actuar en consecuencia.
Dadas las necesidades expuestas, el código fuente de la clase puede ser el que se muestra a continuación.
